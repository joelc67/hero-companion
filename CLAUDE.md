## Design principles (from Joel)

- One shared pool of game-engineered rules, but **every champion/build is unique unto itself** вҖ” the planner must not treat builds as one-size-fits-all within that rule scope. (Recorded at the top of docs/build-doctrine.md.)
- Audits/checkmarks are not ground truth вҖ” Joel's eyes outrank green audits. Coherence audits must hard-fail on empty slots, icon-less pieces, and validation noise.
- **Role doctrine**: CoH is a role-based game first. The Role picker is the declared objective; off-role is only ever an explicit user choice (warn, don't block). Support/control roles follow the **invisible-role doctrine**: their output must be exceptionally powerful just to be noticed вҖ” maximize debuff magnitude Г— uptime, control reach, sustain; never generic damage slotting on a non-damage role.
- **Optimization doctrine ("3D chess")**: think to the END вҖ” converge with restarts and honest certificates, never truncated-as-done; explore, don't prune; **NO ban lists** ("This picker is not a child") вҖ” when the search picks trash, fix the model term that made trash look good; LEARN across runs (champions/marginals/lessons/retrospective); masters are evidence and a floor to beat, never prescriptions ("evolve BEYOND master setups").
- **Wiki-verify / GAME-FIRST**: never build on best guesses. Order of authority: game client bins (`C:\Games\HC2\assets\live\` via Bin Crawler/Pigg Wrangler in tools/gamedata) вҶ’ dev-archive docs вҶ’ measured logs вҶ’ wiki paste from Joel (wikis block WebFetch) вҖ” never fan posts. When the tool and the game disagree, the game is right.
- **Universal rules, no hacks**: a game-rule fix is implemented archetype-independently and proven with an all-AT audit (audit_epic_tiers / audit_slot_schedule pattern) вҖ” never patch just the reported case.
- **Harden-before-certify rule (Joel, 2026-07-08)**: no long certification run (champion refresh, release battery, gold-standard regeneration) STARTS while a correctness question is open or a verification improvement is pending. Certification runs are expensive to invalidate; verification changes are cheap to land вҖ” sequence cheap-and-clarifying before expensive-and-committing, always. (Context: the v28 refresh launched before the coverage-denominator retrofit, which then surfaced questions that put a running 2-hour refresh at risk for the third time in one day.) When a question surfaces mid-run: model/data-affecting вҶ’ stop the run immediately; out of scope вҶ’ make the checker state the exclusion explicitly so it is never rediscovered.
- **Sharpened harden-before-certify (Joel + Cowork, 2026-07-14): roster EXPANSION is certification вҖ” the rule applies to it in full. No new champion certification waves start while ANY objective-correctness question is open.** (Context: Maelwys's aspect-ordering suspicion was on file before most of the weekend's convergence hours ran; the wave should have paused. Work order A is the critical path until resolved; in-flight convergences may finish as sunk compute/knowledge, but their RELEASE holds for A's verdict unless Joel says otherwise, and evaluate-first then covers every certified context against A's fix.)
- **Coverage-denominator rule (Joel, 2026-07-08)**: every audit/reality check/battery prints **"N of M expected checked"** where M comes from an independent source (game snapshot count, data enumeration done OUTSIDE the check loop, or a pinned constant) вҖ” and **hard-fails when N < M**. A checker that can't state its denominator can silently lie (twice in one day: the coherence audit never examined icons, the set-bonus check silently verified 43 of 282 values). Retrofitted 2026-07-08 into demo_single_build_fixes, audit_slotting_coherence/slot_legality/slot_schedule/epic_tiers, reality_check_setbonuses/procs/gamedata/powers/converter.
- **Boost boundary (Joel, 2026-07-09)**: champions are NEVER built, certified, or scored on boosted (+1..+5) enhancement values вҖ” certify clean; boosting appears only as a display preview (queued feature) and later a priced upgrade-ladder rung. If solver-boosting work ever proposes certifying boosted champions, the answer is no.
- **Choice doctrine**: "People should always have a choice." Anything touching user data/machines = informed opt-in, remembered + reversible "no", advise-don't-override; sharing consent separate from local capture; anonymous by default; NO PII ever.
- **Custom-targets doctrine (Joel's ruling, 2026-07-20): HONOR the user's stated goal вҖ” and STATE what it costs.** "If the end user wants to reach a goal, as long as we state what might be lost or changed, we honor their request." Custom targets are the objective's priority at BOTH layers (the pick proposal must serve the ask вҖ” the Tough Hide case вҖ” not just the slotting ILP), the tradeoff is stated in numbers (what was given up: damage/bonuses/powers swapped), and only the genuinely unreachable is refused вҖ” with numbers and a remedy ("best reachable: 42.1; Tough Hide adds ~5 вҖ” add it?"). The editor states its drop-semantics ("cleared = no ask; your kit's innate stats still count"). Never silently traded. Field origin: the 7/20 forum report ("do what I told you, not what you think I want" вҖ” his philosophy IS our choice doctrine); the user's two cases (Inv Tanker all-def-45 / fire-res-90-only) are the standing acceptance pins.
- Copy rules: never say "illegal" bare ("the game won't allow it" instead); plain English before jargon; "easiest route, never the only way"; **no em dashes in outward-facing text Joel sends** (Discord/forum вҖ” "dead AI giveaway").

## Certification protocol rules (learned the hard way вҖ” each one has a corpse behind it)

- **EXPENSIVE RUNS LAUNCH DETACHED, NEVER AS SESSION BACKGROUND TASKS (2026-07-16).** Any convergence wave / champion build-out / roster battery starts as its OWN process that outlives this chat session вҖ” a `.bat` launched via `Start-Process -WindowStyle Hidden` (the `Run E Overnight.bat` / `Run V33 Wave.bat` pattern), logging to a file. **Why:** the first v33 wave ran as session-owned background tasks, and a chat-session **auth failure** (13:23вҖ“13:24 UTC) tore the session down and killed both live workers mid-context вҖ” Battle_Axe at sweep 20, farm_active at sweep 14, ~50вҖ“55 min of compute each, gone. deep_optimize does NOT checkpoint mid-context, so an interrupted context restarts from scratch; only COMPLETED contexts are banked in their shard, so the loss is real. Session background tasks are for short probes and watchers ONLY. **Launch mechanism (2026-07-16 evening): Windows scheduled task, not `Start-Process`** (Start-Process detachment failed to survive twice on 7/16). Register via `Register-ScheduledTask` cmdlets вҖ” `schtasks.exe` quoting breaks under PS 5.1. Known behaviors: BOTH visible-console generations died to a literal `^C` in their logs (gen 1 at ~34s, gen 2 at ~10 min вҖ” visible task consoles ARE killable on this box; killer unidentified, Joel asked). **The working shape: `launch_hidden.vbs` вҖ” the task action runs `wscript.exe launch_hidden.vbs "<bat name>"`, window style 0, no console visible, nothing to close.** S4U ("run whether logged on or not") would also work but is **Access-denied without elevation** (needs Joel at an admin prompt). Unregister the task after the processes are confirmed running вҖ” the orphaned process tree keeps running, and nothing stale re-arms on the next boot (the 7/15 disarm rule by construction). Recovery when it happens anyway: relaunch (detached) вҖ” `converge_parallel` skips already-certified shards, so completed contexts resume for free вҖ” then **verify the shard logs are actually advancing** (two snapshots, sweep lines appearing) before trusting it, and RE-LAUNCH the 5080 dev server if it was session-owned too. вҡ  **Watch the shard-prefix collision:** a resumed wave with fewer workers reassigns `_pN` suffixes, so a completed shard (e.g. `champions_shard_v33_p1.json` holding farm_afk) can be OVERWRITTEN by a different context landing on the same worker index вҖ” copy the completed shard to a non-colliding name (still `champions_shard_*` so the union keeps it) BEFORE relaunching.
- **NEVER MERGE A SHARD WHOLESALE вҖ” MERGE BY CONTEXT, AND CHECK THE VERDICT FIRST (2026-07-16, two near-misses in one hour).** `converge_parallel` prints a "merge when ready:" hint listing every shard it wrote. **That hint is a footgun and must not be followed blindly:** (a) a worker APPENDS to an existing shard file, so one shard can hold a context you are shipping AND one that is HELD вҖ” the v33 p1 shard held farm_afk (merging) next to farm_active (held on a failed acceptance case), and the printed command would have shipped the held champion; (b) `--replace` supersedes the incumbent by construction, so a re-convergence that is WORSE silently replaces a better build вҖ” evaluate-first caught exactly that (re-converged canonical 177.6 / sustain 39.8 vs incumbent 186.9 / 42.5; the incumbent was restored). **Therefore: before any merge, read the shard's CONTEXTS (not just its filename), merge only the cleared ones (split a mixed shard by copying the wanted context to its own file), and after merging run evaluate-first and KEEP THE CANONICAL WINNER вҖ” a recert earns supersession, it is not entitled to it.** A held context's shard is then renamed OUT of the glob so its re-convergence is never skipped.
- **SHARD RETIREMENT AT MERGE (2026-07-16).** The moment a shard's contents merge into champions.json, RENAME it to a non-matching suffix вҖ” `champions_shard_X.json.merged_YYYY-MM-DD` (generalising the `.held_pricing_ruling` pattern) вҖ” so no glob can ever pick a stale copy up again. **Why:** `certified_union()` globs `champions_shard_*.json` at the repo root and a stale copy SHADOWED a live champions.json entry: the merged farm_p0 shard (canonical_score: None) overwrote the annotated farm_afk entry, so evaluate_first read old=None and printed **BASELINE for a champion whose canonical had genuinely MOVED 169.0 вҶ’ 186.9**. Measured: **8 stale shards, 28 shadowing attempts.** It was caught by Joel's explicit "farm_afk re-opened" order, NOT by protocol вҖ” i.e. luck. The `certified_union` stale-shadow guard (a shard may add contexts or supersede an entry with no canonical baseline, but may NEVER erase one that exists) is the second lock. Belt and suspenders. **Shards that are deliberately never merged are NOT stale and must NOT be retired** вҖ” the E ground truths (`champions_shard_e_gt_*.json`) are the standing benchmark suite and must stay in the union so evaluate-first keeps re-verifying them.
- **VERDICT BEFORE `--write` (2026-07-16).** Read and RECORD the wave's verdict before `evaluate_first --write` runs. `--write` overwrites `canonical_score` вҖ” i.e. the very values the verdict is computed against вҖ” so running it first destroys the comparison. On 2026-07-16 the verdict survived only because champions.json was committed and the old baselines were recoverable from `git show HEAD:benchmarks/champions.json`. Recovery worked; the order of operations is now a rule, not a habit.
- **EVERY NEW PRICING TERM SHIPS WITH A NEGATIVE CONTROL** (2026-07-16) вҖ” a real build that must read **0.0** вҖ” alongside its positive test. v33's ruling C is the template: the negative control (Maelwys's own farm builds, which slot Superior *Brute's* Fury and NOT Unrelenting Fury) is what proved the term reads ACTUAL slotting rather than firing on a set-name lookalike; the positive test only proves the arithmetic.
- **"READY FOR YOUR WALK" IS A CLAIM WITH A DEFINITION вҖ” AND IT NAMES THE COMMIT HASH (2026-07-16).** The words mean ALL of: (a) the server process was restarted from current HEAD *after* the last server-side edit, (b) a FRESH page load with **zero injected state**, (c) the exact URL Joel opens, driven through a REAL entry path he would use (entry card вҶ’ wizard/Resume вҖ” not a hand-built `build` object), and (d) the claim **states the commit hash the server is running**. Anything less is "probably ready" and must be said that way. **Why:** on 2026-07-16 I reported "5080 is ready for your final acceptance walk вҖҰ browser-verified pop-ups", his walk found nothing, and the reconstruction was damning вҖ” every one of my "browser-verified" runs had done `build.powers = ap.powers; renderPowers(); await recompute();`, i.e. **I hand-constructed the build object and called the render functions directly. I never once clicked an entry card, ran the wizard, or resumed a build.** I verified a code path I built, not the app he uses. (The server WAS current and the panel DOES work via Resume вҖ” so the feature was fine and the claim was still false. That is the point: it was luck, not verification. Had the real path been broken, "verified" would have shipped a feature nobody could reach.) Verification theater is the same defect class as a fake progress bar вҖ” and the tell is identical: the check never touched the thing the user touches.
- **THE MACHINE CLOCK IS LOCAL EASTERN вҖ” `Get-Date` IS THE TIME AUTHORITY, NEVER A TZ CONVERSION (2026-07-16).** This box's clock already reads Eastern. `TZ=America/New_York date` in the bash tool re-interprets that Eastern time AS UTC and shifts it +4h вҖ” a pure fiction. **Measured cost:** code ran that "cross-check", believed 12:28 PM EDT was "16:28 ET", concluded the 4:05 ritual had passed, and told Joel the day's work carried to tomorrow вҖ” with three hours of afternoon left. Same family as yesterday's date-erratum (entries stamped 7/16 while the box said 7/15). **Rule: for anything a deadline, ritual, or report timestamp depends on, run `Get-Date` in PowerShell and use it verbatim.** `date -u` is fine for UTC-labelled facts (log correlation), but never convert between them to derive local time, and never let a derived time override the machine's own answer.
- **SCRIPTED-WRITE GUARD (2026-07-16) вҖ” the catch is mechanical, not vigilant.** Three scripted-write incidents in ONE day (powers.json pretty-printed by `json.dump(indent=1)` into a 40MB reformat; CLAUDE.md flipped LFвҶ’CRLF by Python's Windows text-mode write; and the PS5.1 ANSI-mangling item already on record). Each was caught by *noticing the diff looked too big* вҖ” a good instinct is not a control. So:
  - **(a) Every scripted edit of a repo file writes in binary / newline-preserving mode** (`open(p,'rb')` вҶ’ transform вҶ’ `open(p,'wb')`), and matches the file's existing serialisation (powers.json is a COMPACT single line вҖ” `json.dump(data, f)`, never `indent=`; check what the sibling patchers do before inventing a format).
  - **(b) Before committing any scripted edit, compare diff size to INTENDED size.** `git diff --stat` vs what you meant to change; cross-check with `git diff --ignore-all-space --stat` вҖ” if insertions+deletions exceed the intent by more than ~2Г—, or the whitespace-blind diff is dramatically smaller than the raw one, **STOP and diagnose before committing.** A documentation edit reporting a whole-file diff IS a whole-file edit until proven otherwise.
  - **(c) Prefer the dedicated edit tooling over hand-rolled rewrite scripts** for source/docs вҖ” a scripted rewrite is for when there is no alternative, not for convenience. **NEVER use PowerShell string rewrites on source files** (PS5.1 reads BOM-less UTF-8 as ANSI and mangles unicode вҖ” already on record).
